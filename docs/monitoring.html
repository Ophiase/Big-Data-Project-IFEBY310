  1. Relation does not differ between Analyzed and Optimized Logical Plan. The query optimizer in the RDBMS did not find any opportunities to further optimize the logical plan after the initial analysis phase.

  2. The Physical Plan differs from the Optimized Logical Plan in that it includes the ColumnarToRow operator, which is a Spark-specific operator that converts data from the columnar format used for storage and processing to the row format used for execution. The FileScan parquet operator is also specific to Spark and is used to scan Parquet data files.

    • It requires :
      • One stage to read the parquet file of travels.
      • One stage to show it.
      • Two stages (over two jobs) to show the df_star
    • HashAggregate efficiently performs aggregations by grouping data using hash tables, optimizing memory usage and processing speed.
    • Exchange hashpartitioning redistributes data across nodes based on hashed keys, enabling parallel processing and necessary data colocation for operations like joins, albeit with the cost of shuffle operations.

  4. The above operations do not perform shuffle operations. But, neverthless, if we look down on jobs performing at .toPandas operation, they do perform a shuffle operation to do so.

    • Tasks are the smallest units of work within a stage, each processing a single data partition.
